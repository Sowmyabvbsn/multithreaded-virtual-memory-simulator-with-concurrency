Introduction & Motivation

Background
Context
  • Operating Systems are a cornerstone of computer science education.
  • Concepts like demand paging, thread scheduling, and deadlocks are abstract and hard to grasp from textbooks alone.
  • Existing simulation tools are either too simplistic or require advanced setup.
  • There is a need for an integrated, visual, interactive learning platform.
Motivation
  • Provide a hands-on environment for OS algorithm exploration.
  • Bridge the gap between theory and practice through real-time visualization.
  • Leverage AI assistance for intelligent dataset generation and analysis.
Key Challenges Addressed
  • Visualizing page replacement frame transitions step by step
  • Simulating concurrent thread execution safely in a single-machine environment
  • Demonstrating race conditions, deadlocks, and priority inversion
  • Comparing multiple algorithms on identical workloads simultaneously
  • Generating intelligent benchmark datasets using generative AI

Page Replacement Algorithms

Core Algorithms

FIFO — First In First Out

Evicts the page that arrived first in memory. Implemented using a FIFO queue. Simple but suffers from Belady's Anomaly. Best for sequential access patterns.

LRU — Least Recently Used

Evicts the page not accessed for the longest time. Uses list reordering on each access. Industry standard for general workloads. No Belady's Anomaly. Excellent temporal locality support.

MRU — Most Recently Used

Evicts the most recently used page — counter-intuitive but effective for cyclic scan patterns. Educational comparison point showing locality assumptions can be reversed.

OPT — Optimal Algorithm

Evicts the page that will not be used for the longest time in the future. Theoretical best — requires future knowledge so not implementable in practice. Serves as the performance upper bound for comparison.

Comparative Performance (Reference: 7 0 1 2 0 3 0 4 2 3 0 3 2, 3 Frames):  OPT = 6 faults • LRU = 7 faults • MRU = 9 faults • FIFO = 9 faults

Thread Scheduling Algorithms

Scheduling

FCFS

First-Come-First-Served
  • Non-preemptive simple FIFO queue
  • Threads execute in arrival order
  • Low overhead, easy to implement
  • Suffers from Convoy Effect
  • Best for batch processing workloads
Non-preemptive

Round-Robin

Time-Sliced Execution
  • Preemptive with configurable time quantum (1–10)
  • Fair CPU allocation across all threads
  • Good average response time
  • Context switches increase with smaller quantum
  • Ideal for time-sharing systems
Preemptive Fair

Priority-Based

Highest Priority First
  • Selects thread with highest priority from ready queue
  • Flexible, supports real-time scenarios
  • Can cause starvation of low-priority threads
  • Demonstrates priority inversion with Mutex
  • Ideal for real-time systems
Priority-driven Starvation Risk
Key Insight: Round-Robin with a quantum equal to the total burst time degenerates into FCFS. Smaller quantum improves fairness but increases context-switch overhead, demonstrating the fundamental trade-off in time-sharing design.

Synchronization Mechanisms

Concurrency

Mutex — Mutual Exclusion

  • Binary lock: only one thread holds it at a time
  • Thread blocked if lock is held; added to waiting queue
  • On release, first waiting thread is automatically granted access
  • Demonstrates exclusive access to shared memory regions
  • Can create deadlock when multiple mutexes are used
T1 acquires Mutex-A → T2 waits T2 acquires Mutex-B → T1 waits for Mutex-B ⇒ DEADLOCK: Circular wait detected

Semaphore — Counting Semaphore

  • Configurable permits (1–5); multiple threads can proceed simultaneously
  • Permit decremented on acquire, incremented on release
  • Waiting queue used when all permits exhausted
  • Models resource pools (e.g., limited DB connections)
  • Reduces contention vs Mutex for parallel workloads
Semaphore(2): T1 acquires → permits=1 T2 acquires → permits=0 T3 waits → blocked T1 releases → T3 unblocked
Thread States & Transitions
NEW → READY → RUNNING → COMPLETED ↕ ↕ BLOCKED ←← lock wait WAITING ←← resource unavailable
Race Condition Demo (No Sync)

When synchronization is set to None, multiple threads access the same pages without coordination. The simulator shows interleaved accesses, demonstrating how unsynchronized concurrent access leads to unpredictable memory states — the classic race condition problem.

Deadlock Detection

Safety
Deadlock Conditions (Coffman, 1971)
  • Mutual Exclusion: Resources held exclusively
  • Hold and Wait: Thread holds one resource while waiting for another
  • No Preemption: Resources not forcibly taken
  • Circular Wait: T1 waits for T2, T2 waits for T3, T3 waits for T1
Detection Algorithm
  • Builds a Resource Allocation Graph (RAG) at each simulation step
  • Nodes: threads and lock resources; Edges: holds / waiting-for
  • Depth-First Search (DFS) detects cycle in the graph
  • All threads in the cycle are flagged as deadlocked
  • Simulation halts with visual alert and thread list
RAG Example — Deadlock Scenario
T1 holds Mutex-A → waits for Mutex-B T2 holds Mutex-B → waits for Mutex-A T3 holds Mutex-B → waits for Mutex-A DFS traversal: Visit T1 → follow Mutex-B → T2 Visit T2 → follow Mutex-A → T1 CYCLE DETECTED: [T1, T2] ⇒ DEADLOCK REPORTED
Educational Value: Students observe the exact moment deadlock occurs, see which threads are frozen, and understand how to prevent it by enforcing a consistent lock-ordering protocol.
