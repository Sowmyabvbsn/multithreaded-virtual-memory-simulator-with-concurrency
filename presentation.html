Page Replacement Algorithms

Core Algorithms

FIFO

First In First Out
  • Evicts the page that arrived first in memory
  • Implemented using a LinkedList queue
  • Simple, low overhead
  • Suffers from Belady's Anomaly (more frames can cause more faults)
  • Best for sequential access patterns
  • Eviction: O(1)

LRU

Least Recently Used
  • Evicts the page not accessed for the longest time
  • Uses list reordering on each access
  • Industry standard for general workloads
  • No Belady's Anomaly
  • Excellent temporal locality support
  • Eviction: O(n)

MRU

Most Recently Used
  • Evicts the most recently used page
  • Counter-intuitive but effective for cyclic scan patterns
  • Educational comparison point
  • Shows locality assumptions can be reversed
  • Eviction: O(1)

OPT

Optimal Algorithm
  • Evicts page that will not be used for longest time in future
  • Theoretical best — requires future knowledge
  • Not implementable in practice
  • Serves as performance upper bound for comparison
  • Uses lookahead scan
  • Eviction: O(n)
Comparative Performance (Reference: 7 0 1 2 0 3 0 4 2 3 0 3 2, 3 Frames):  OPT = 6 faults • LRU = 7 faults • MRU = 9 faults • FIFO = 9 faults — LRU closely approximates OPT.

Belady's Anomaly — FIFO Demonstration

Key Finding
What is Belady's Anomaly?

For some page replacement algorithms, increasing the number of frames can increase the number of page faults. This counter-intuitive result was proved by Belady et al. (1969) and only affects non-stack algorithms like FIFO.

  • FIFO is not a stack algorithm — the set of pages in memory with n frames is not necessarily a subset of the set with n+1 frames
  • LRU and OPT are stack algorithms — they are immune to Belady's Anomaly
  • More frames always means fewer or equal faults for stack algorithms
Why It Matters Educationally
  • Students can observe the anomaly firsthand by running FIFO with 3 vs 4 frames
  • Challenges the intuition that "more memory = better performance"
  • Demonstrates the importance of algorithm selection in OS design
  • Our simulator provides the comparison tool to verify this instantly
Demonstration in Our Simulator
Reference: 1 2 3 4 1 2 5 1 2 3 4 5 FIFO with 3 frames: Faults: 9 FIFO with 4 frames: Faults: 10 (ANOMALY!) More frames, MORE faults! LRU with 3 frames: Faults: 10 LRU with 4 frames: Faults: 8 (Normal: fewer faults) Conclusion: FIFO is not a stack algorithm. LRU is a stack algorithm - no anomaly.
Key Takeaway: Belady's Anomaly proves that algorithm choice matters more than simply adding memory. LRU's stack property guarantees monotonic improvement with more frames.

Thread Scheduling Algorithms

Scheduling

FCFS — First-Come-First-Served

  • Non-preemptive FIFO queue scheduling
  • Threads execute in arrival order
  • Low overhead, easy to implement
  • Suffers from Convoy Effect — short jobs wait behind long jobs
  • Best for batch processing workloads
  • No context switches within a thread's burst
Non-preemptive

Round-Robin

  • Preemptive with configurable time quantum (1–10)
  • Fair CPU allocation across all threads
  • Good average response time
  • Context switches increase with smaller quantum
  • Ideal for time-sharing systems
  • Quantum tracked per page access
Preemptive Fair

Priority-Based

  • Selects thread with highest priority (1–10) from ready queue
  • Flexible, supports real-time scenarios
  • Can cause starvation of low-priority threads
  • Demonstrates priority inversion when combined with Mutex
  • Ideal for real-time systems
  • O(n) scan for highest priority
Priority-driven Starvation Risk
Key Trade-off: Round-Robin with quantum equal to total burst time degenerates into FCFS. Smaller quantum improves fairness but increases context-switch overhead. Priority scheduling gives best avg wait for heterogeneous workloads but risks starvation.

Synchronization Mechanisms

Concurrency

Mutex — Mutual Exclusion Lock

  • Binary lock: only one thread holds it at a time (1 permit)
  • Thread blocked if lock is held; added to FIFO waiting queue
  • On release, first waiting thread is automatically granted access and set to READY
  • Demonstrates exclusive access to shared memory regions
  • Can create deadlock when multiple mutexes are acquired in different orders
  • Tracks acquisition count for statistics
T1 acquires Mutex-A -> T2 waits (BLOCKED) T2 acquires Mutex-B -> T1 waits for Mutex-B (BLOCKED) => DEADLOCK: Circular wait detected by DeadlockDetector

Semaphore — Counting Semaphore

  • Configurable permits (1–5); multiple threads can proceed simultaneously
  • Permit decremented on acquire, incremented on release
  • FIFO waiting queue used when all permits exhausted
  • Models resource pools (e.g., limited DB connections, buffer slots)
  • Reduces contention vs Mutex for parallel workloads
  • On release, dequeues one waiting thread and sets to READY
Semaphore(2): T1 acquires -> permits=1 T2 acquires -> permits=0 T3 waits -> BLOCKED T1 releases -> T3 unblocked, permits=0
Thread States & Transitions
READY -> RUNNING -> COMPLETED ^ | ^ | v | BLOCKED <- WAITING READY: Thread is eligible to run RUNNING: Currently executing page access WAITING: Waiting for resource (lock/semaphore) BLOCKED: Lock acquisition failed, in wait queue COMPLETED: All page references processed
Lock Release Behavior
  • After a thread accesses a page, there is a 50% probability of releasing held locks
  • This probabilistic release creates realistic contention patterns
  • When no synchronization is selected (None), threads access shared frames without coordination — demonstrating race conditions
LockResource Implementation
  • All methods are synchronized for thread safety
  • tryAcquire(): Non-blocking — returns true if permit available, false + enqueue if not
  • release(): Increments permits, dequeues and unblocks next waiter
  • forceRelease(): Deadlock resolution — resets all permits, unblocks all waiters

Deadlock Detection

Safety
Coffman's Four Conditions (1971)
  • Mutual Exclusion: Resources held exclusively by one thread at a time
  • Hold and Wait: Thread holds one resource while waiting for another
  • No Preemption: Resources cannot be forcibly taken from a thread
  • Circular Wait: T1 waits for T2, T2 waits for T3, T3 waits for T1

All four conditions must hold simultaneously for deadlock to occur.

Detection Algorithm (DeadlockDetector.java)
  • Builds a Resource Allocation Graph (RAG) at each simulation step
  • Nodes: ProcessThreads in BLOCKED state + Lock resources
  • Edges: Thread A -> Thread B if A waits for a lock held by B
  • Depth-First Search (DFS) detects cycle in the graph
  • Uses recursion stack to detect back edges
  • All threads in the cycle are flagged as deadlocked
  • Simulation halts with visual alert and thread list
  • visualizeGraph() produces text representation of the RAG
RAG Example — Deadlock Scenario
T1 holds Mutex-A, waits for Mutex-B T2 holds Mutex-B, waits for Mutex-A T3 holds Mutex-B, waits for Mutex-A Resource Allocation Graph: T1 --holds--> Mutex-A T1 --waits--> Mutex-B <--holds-- T2 T2 --waits--> Mutex-A <--holds-- T1 T3 --waits--> Mutex-A DFS traversal: Visit T1 -> follow wait for Mutex-B -> T2 Visit T2 -> follow wait for Mutex-A -> T1 Back edge detected: T1 already in recursion stack CYCLE: [T1, T2] => DEADLOCK REPORTED => Simulation halts
Deadlock Resolution
  • When deadlock is detected, LockResource.forceRelease() is called
  • Resets all permits to max, unblocks all waiting threads
  • Simulation can continue after resolution
  • Educational: Students learn to prevent deadlock via consistent lock ordering
